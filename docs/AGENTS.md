###### **Estándares de Código**



1\. Idioma: Todo el código deberá ser escrito en inglés, independientemente del contexto del proyecto o idioma de la UI.

2\. Calidad del Código:



· Estructura organizada, optimizada y clara.

· Directorios y jerarquías de archivos altamente organizados.

· Sin comentarios o documentación a menos que se solicite explícitamente.

· Seguir las convenciones establecidas para el lenguaje/framework.

· Preferir claridad sobre ingeniosidad.

· Eliminar código muerto y lógica redundante.



3\. Rendimiento y Eficiencia:



· Minimizar tamaños de bundle y tiempos de carga.

· Optimizar algoritmos y estructuras de datos.

· Usar lazy loading y code splitting cuando sea aplicable.

· Cachear apropiadamente

· Evitar re-renders o llamadas a funciones innecesarias.



4\. Convenciones de Nombres:



· Nombres claros y descriptivos para variables, funciones y clases.

· Consistencia en toda la base de código.

· Usar terminología específica del dominio con precisión.



###### **Estilo y Personalidad**



· Escribir de forma profesional, pulida y precisa.

· Comunicarse eficientemente sin verbosidad innecesaria.

· Mantener consistencia con patrones existentes en el proyecto.

· Evitar soluciones genéricas o basadas en plantillas.



###### **Estructura del Proyecto**



Todo proyecto debe incluir la siguiente estructura.



raíz/

├── essentials/

│   ├── AGENTS.md (Copia exacta de este archivo)

│   └── CHANGELOG.md (Registro detallado de todos los cambios)

├── src/

├── \[Otros directorios según sea necesario]

└── \[Archivos de configuración]



###### **Requisitos de CHANGELOG.md**



El archivo CHANGELOG.md debe hacer lo siguiente.



· Registrar absolutamente todos los cambios realizados en el proyecto.

· Usar un formato claro y con marca de tiempo.

· Incluir descripciones de qué cambió y por qué.

· Actualizarse continuamente, no retrospectivamente.

· Servir como registro de auditoría completo.



Ejemplo de Formato:



\# \[DD-MM-YYYY - HH:MM]



· Descripción del cambio.

· Otro cambio.



###### **Herramientas y Dependencias**



· Usar librerías establecidas y bien mantenidas.

· Preferir dependencias mínimas cuando sea posible.

· Documentar cualquier utilidad o helper personalizado.

· Mantener dependencias actualizadas.



###### **Testing y Aseguramiento de Calidad**



· Escribir código testeable.

· Seguir mejores prácticas específicas del lenguaje.

· Validar entradas apropiadamente.

· Manejar errores elegantemente.



###### **Control de Versiones**



· Mensajes de commit claros y descriptivos en español. (El idioma puede variar dependiendo del proyecto)

· Commits lógicos y atómicos.

· Mantener el historial legible y significativo.



###### **Documentación**



· El código debe autodocumentarse cuando sea posible pero solo en caso de que sea un proyecto open-source y se vaya a subir a alguna plataforma como GitHub.

· Proporcionar README.md para instrucciones de instalación y uso en caso de que se vaya a publicar el código.

· Las firmas de API/funciones deben ser claras.

· Solo agregar comentarios inline para lógica no obvia
